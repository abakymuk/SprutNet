# Руководство по миграции базы данных SprutNet

## Обзор

Это руководство описывает процесс применения унифицированной схемы базы данных для SprutNet Shipping Planner, которая объединяет:

- **Prisma модель Maersk API** (нормализованная структура)
- **Расширенные пользовательские функции** (кэширование, аналитика, мониторинг)
- **Система управления конфигурацией** (feature flags, настройки)

## Варианты миграции

### Вариант 1: Полная миграция (рекомендуется для новых проектов)

Если у вас новый проект или вы готовы полностью пересоздать базу данных:

1. **Проверьте текущее состояние базы данных**
   ```sql
   -- Выполните скрипт проверки
   -- supabase/check_current_state.sql
   ```

2. **Очистите существующую базу данных**
   ```sql
   -- Выполните безопасный скрипт очистки
   -- supabase/safe_cleanup.sql
   ```

3. **Примените унифицированную схему**
   ```sql
   -- Выполните миграцию
   -- supabase/migrations/20250101000002_unified_schema.sql
   ```

### Вариант 2: Поэтапная миграция (для существующих проектов)

Если у вас уже есть данные, которые нужно сохранить:

1. **Создайте резервную копию**
   ```bash
   # Экспорт текущей базы данных
   pg_dump $DATABASE_URL > backup_$(date +%Y%m%d_%H%M%S).sql
   ```

2. **Примените миграции по порядку**
   ```sql
   -- 1. Создайте новые таблицы (без удаления существующих)
   -- supabase/migrations/20250101000002_unified_schema.sql
   ```

3. **Перенесите данные** (если необходимо)
   ```sql
   -- Скрипты миграции данных будут созданы отдельно
   ```

## Пошаговая инструкция

### Шаг 1: Подготовка

1. **Проверьте подключение к Supabase**
   ```bash
   # Убедитесь, что у вас есть доступ к проекту
   supabase status
   ```

2. **Создайте резервную копию** (обязательно!)
   ```bash
   # В Supabase Dashboard -> SQL Editor
   -- Выполните экспорт всех данных
   ```

### Шаг 2: Диагностика текущего состояния

1. **Откройте Supabase Dashboard**
   ```
   https://supabase.com/dashboard/project/[YOUR_PROJECT_ID]
   ```

2. **Перейдите в SQL Editor**

3. **Выполните диагностику**
   ```sql
   -- Скопируйте содержимое файла:
   -- supabase/check_current_state.sql
   -- И выполните в SQL Editor
   ```

4. **Проанализируйте результаты**
   - Если таблиц 0: база пуста, можно применять миграцию
   - Если таблиц 1-25: частично заполнена, проверьте структуру
   - Если таблиц >25: содержит данные, создайте резервную копию

### Шаг 3: Очистка (если необходимо)

1. **Выполните безопасную очистку**
   ```sql
   -- Скопируйте содержимое файла:
   -- supabase/safe_cleanup.sql
   -- И выполните в SQL Editor
   ```

2. **Проверьте результат очистки**
   - Скрипт покажет количество оставшихся таблиц
   - Если осталось 0 таблиц: очистка успешна
   - Если остались таблицы: проверьте список

### Шаг 4: Применение миграции

1. **Выполните миграцию**
   ```sql
   -- Скопируйте содержимое файла:
   -- supabase/migrations/20250101000002_unified_schema.sql
   -- И выполните в SQL Editor
   ```

2. **Проверьте результат**
   - Должно появиться сообщение "Unified schema migration completed successfully!"
   - Количество таблиц должно быть 25

### Шаг 5: Проверка

1. **Проверьте создание таблиц**
   ```sql
   SELECT table_name 
   FROM information_schema.tables 
   WHERE table_schema = 'public'
   ORDER BY table_name;
   ```

2. **Проверьте индексы**
   ```sql
   SELECT indexname, tablename 
   FROM pg_indexes 
   WHERE schemaname = 'public'
   ORDER BY tablename, indexname;
   ```

3. **Проверьте RLS политики**
   ```sql
   SELECT schemaname, tablename, policyname 
   FROM pg_policies 
   WHERE schemaname = 'public'
   ORDER BY tablename, policyname;
   ```

4. **Проверьте представления**
   ```sql
   SELECT viewname 
   FROM pg_views 
   WHERE schemaname = 'public'
   ORDER BY viewname;
   ```

### Шаг 6: Настройка

1. **Проверьте начальные данные**
   ```sql
   -- Проверьте страны
   SELECT COUNT(*) FROM countries;
   
   -- Проверьте коды перевозчиков
   SELECT COUNT(*) FROM carrier_codes;
   
   -- Проверьте feature flags
   SELECT * FROM feature_flags;
   
   -- Проверьте конфигурацию
   SELECT * FROM system_config;
   ```

2. **Настройте планировщик очистки кэша**
   ```sql
   -- В Supabase Dashboard -> Database -> Functions
   -- Создайте cron job для очистки кэша
   SELECT cron.schedule('cleanup-cache', '*/5 * * * *', 'SELECT cleanup_expired_cache();');
   ```

## Безопасность

### Row Level Security (RLS)

Все пользовательские таблицы защищены RLS политиками:

- **Пользователи**: Доступ только к своим данным
- **Maersk данные**: Чтение для всех аутентифицированных пользователей
- **Системные таблицы**: Доступ только для service_role

### Политики доступа

```sql
-- Пользователи могут видеть только свои данные
CREATE POLICY "Users can view own data" ON users
    FOR SELECT USING (auth.uid() = id);

-- Maersk данные доступны для чтения всем аутентифицированным
CREATE POLICY "Allow authenticated read access" ON vessels 
    FOR SELECT TO authenticated USING (true);

-- Системные таблицы только для service_role
CREATE POLICY "System tables for service role" ON performance_metrics
    FOR ALL USING (auth.role() = 'service_role');
```

## Производительность

### Индексы

Схема включает оптимизированные индексы для:

- **Поиск по кодам**: IMO, UN/LOCODE, carrier codes
- **Полнотекстовый поиск**: Названия судов и портов
- **Временные запросы**: Даты отправления/прибытия
- **JSON данные**: Параметры поиска, конфигурация

### Кэширование

- **TTL кэш**: Автоматическое истечение данных
- **LRU эвикция**: Удаление редко используемых записей
- **Сжатие**: Для больших объектов
- **Статистика**: Отслеживание hit/miss ratio

## Мониторинг

### Метрики производительности

- **Время ответа API**: Среднее, медиана, 95-й перцентиль
- **Кэш эффективность**: Hit ratio, miss ratio
- **Ошибки**: Количество, типы, детали
- **Использование**: Количество запросов, пользователей

### Аналитика

- **Популярные маршруты**: Топ-10 по поискам
- **Активность пользователей**: Сессии, поиски, время
- **Статистика флота**: Используемые суда, маршруты
- **Статистика портов**: Загруженность, активность

## Устранение неполадок

### Частые проблемы

1. **Ошибка "relation does not exist"**
   ```sql
   -- Проверьте, что миграция выполнена полностью
   SELECT table_name FROM information_schema.tables WHERE table_schema = 'public';
   ```

2. **Ошибка RLS политик**
   ```sql
   -- Проверьте политики
   SELECT schemaname, tablename, policyname FROM pg_policies WHERE schemaname = 'public';
   ```

3. **Медленные запросы**
   ```sql
   -- Проверьте индексы
   SELECT indexname, tablename FROM pg_indexes WHERE schemaname = 'public';
   ```

4. **Ошибка очистки**
   ```sql
   -- Используйте безопасный скрипт очистки
   -- supabase/safe_cleanup.sql
   ```

### Восстановление

1. **Откат к резервной копии**
   ```bash
   psql $DATABASE_URL < backup_YYYYMMDD_HHMMSS.sql
   ```

2. **Частичный откат**
   ```sql
   -- Удалите конкретные таблицы
   DROP TABLE IF EXISTS table_name CASCADE;
   ```
